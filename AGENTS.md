# Repository Guidelines

## Project Structure & Module Organization

This is a small Clojure library that exposes async/await-style helpers on top of `core.async`, with structured ownership-based cancellation.

- `src/com/xadecimal/async_style/impl.clj` contains the implementation.
- `src/com/xadecimal/async_style/protocols.clj` defines extension protocols such as `IntoPromiseChan`.
- `src/com/xadecimal/async_style.clj` is generated by `expose-api`; do not edit it directly.
- `test/com/xadecimal/async_style_test.clj` contains the library tests and doubles as usage documentation.
- `specs/changes/` is reserved for active RFCs and implementation task notes
  for larger design changes.
- `specs/future-ideas.md` tracks unresolved ideas that do not yet have an RFC.
- `specs/archive/` preserves implemented and superseded RFCs, completed task
  notes, and completed future ideas.
- `deps.edn`, `bb.edn`, `build.clj`, and `build.edn` define dependencies, tasks, publishing metadata, and API generation.

## Build, Test, and Development Commands

- `bb test`: runs the full suite against the default `core.async` version and the `:test-1.7` compatibility alias.
- `clojure -M:test`: runs tests once with the default dependency set.
- `bb lint`: runs the build-edn lint task.
- `bb gen`: regenerates `src/com/xadecimal/async_style.clj` from the vars listed in `build.clj`.
- `bb install`: builds and installs the library locally.
- `bb release`: runs lint, API generation, tests, and local install.

Use `bb test` before submitting changes that affect behavior, but prefer running specific tests when it makes more sense to avoid unnecessary test output. If you only modified comments or doc-string, you don't need to re-run tests.

### Running Specific Tests

Use `bb test-vars` to run targeted tests instead of the whole suite. Pass fully qualified symbols without quotes or brackets.

* **Single test:** `bb test-vars namespace.core-test/my-specific-test`
* **Multiple tests:** `bb test-vars namespace.core-test/test-one namespace.core-test/test-two`

## Coding Style & Naming Conventions

Follow idiomatic Clojure formatting: two-space indentation, small focused functions, kebab-case public names, and predicate names ending in `?`. Macros such as `async`, `blocking`, `compute`, `await`, and `wait` should keep behavior consistent across error handling, cancellation, and one-level producer settlement.

Order top-level definitions by dependency and do not use `declare`. When dependencies form a cycle, refactor the boundary to break the cycle and move the minimum number of definitions necessary.

Give every top-level function, macro, protocol, and var a concise docstring. For forms such as `deftype` that do not support docstrings, put a concise documentation comment immediately above the form.

Prefer changing `impl.clj` and `protocols.clj`; regenerate the public API file with `bb gen` when exported vars or docstrings change. Keep comments concise and useful.

## Cancellation & Ownership Semantics

The central rule is: cancellation follows ownership, not observation. Starting `async`, `blocking`, or `compute` inside a running execution creates a direct owned child unless the start happens inside `detach`. Cancelling, failing, or completing a parent cancels unfinished direct children; transitive cancellation happens through each child doing the same.

Awaiting, racing, timing out, or aggregating already-started work does not transfer ownership. `race`, `any`, `all`, `all-settled`, and `timeout` must observe borrowed inputs without cancelling them. Internal watcher work, when needed, should be lightweight and must not make watched inputs owned children.

Producer settlement assimilates one level only for promise-like single-result async values. Nested async-style producers compose because each producer performs that one-level wait before settling. Multi-value source channels, including `async-generator` channels and ordinary raw channels, must be preserved as values; returning a cold generator does not start or own its producer.

Use `detach` for intentional background work that should outlive parent completion or cancellation. Do not introduce a second escape hatch such as `shield` without a new design discussion.

## Async Iteration Semantics

`async-generator` returns a cold core.async-compatible async-style channel. Creating or returning it does not start work; taking from it starts its producer in the current consumption scope. That producer is then owned by that scope and should be cancelled on parent cancellation, failure, or completion.

`yield` publishes raw non-nil values and applies async-style value semantics before putting them on the output channel. Keep generator output lossless: omitted `:buffer-size` means unbuffered pull behavior where code after `yield` waits for the next pull or `areturn`, and positive `:buffer-size` values allow bounded runahead. Do not add dropping/sliding behavior to `async-generator`. Do not wrap yielded values in JS-style `{done?, value}` maps; channel close / nil take means done, so async generators cannot yield nil.

`anext` and `areturn` are the manual lifecycle API. `anext` supports channel-like sources only, starts cold generators in the current consumption scope, and returns one raw value or nil when done. `areturn` requests lifecycle-aware cleanup and waits for finalization; on borrowed plain channels it is a no-op and must not close or cancel them.

`cancel!` on an async-generator channel should delegate to the same lifecycle cleanup path as `areturn` / `async-return!`, but fire-and-forget. It should close the output channel, unblock paused yields, cancel the producer if started, and allow generator `catch` / `finally` to run. `close!` remains only raw channel close and should not be treated or documented as the lifecycle cleanup API.

`adoseq`, `afor`, `areduce`, `atransduce`, and `ainto` are lifecycle-aware consumers. On early exit, reduced completion, errors, or cancellation, they should call generator cleanup and wait for finalization when the source supports it. Borrowed plain channels remain observation-only and must not be closed or cancelled.

Reducing functions and transducer steps run in async-pool orchestration code. They should stay quick and synchronous; blocking, parking, I/O, or heavy compute belongs in `blocking`, `compute`, `adoseq`, `afor`, or an `async-generator` before reduction.

## Testing Guidelines

Tests use `clojure.test` plus `com.xadecimal.testa`. Add tests in `test/com/xadecimal/async_style_test.clj` near the API being changed, using existing `deftest` names such as `timeout-tests` or `cancel!-tests` as the pattern.

Cover both successful values and `Throwable` results for async APIs. For cancellation-sensitive changes, include tests for `cancel!`, `cancelled?`, parent-child propagation, parent completion/failure cleanup, borrowed observation, and `detach` where relevant. Prefer deterministic latches such as raw `core.async` promise channels over sleeps when proving child startup.

## Commit & Pull Request Guidelines

Recent commits use past-tense, concrete summaries, for example `Added structured cancellation...` or `Switched to a callback based implementation...`. Name the affected APIs or behavior and state the user-visible outcome; avoid generic summaries such as `Tightened ... semantics` that identify an area without saying what changed. For multi-part behavioral changes, summarize the main outcomes directly, for example `Prevented race from consuming losing channel values, made all-settled concurrent, and made timeout/time observation-only.` Keep commits focused.

Pull requests should include a concise description, the commands run, and notes about compatibility with `core.async` 1.7 and 1.8+ when relevant. Mention generated-file updates explicitly if `bb gen` changed `src/com/xadecimal/async_style.clj`.

## Agent-Specific Instructions

Do not manually edit the generated public namespace. When changing exported behavior, edit implementation sources first, run `bb gen`, then run tests. Keep README, CHANGELOG, generated docstrings, and tests aligned whenever cancellation, settlement, or `IntoPromiseChan` behavior changes.
